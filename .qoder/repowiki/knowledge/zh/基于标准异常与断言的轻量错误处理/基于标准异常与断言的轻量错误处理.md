本仓库采用 .NET Framework 4.6.1 + WinForms，未引入第三方错误处理框架（如 Polly、FluentValidation、自定义 Result 类型等），整体遵循快速失败加标准异常的轻量策略。

## 1. 系统与方法概览
- 参数校验：在业务类构造函数与方法入口处使用 ArgumentException、ArgumentNullException、ArgumentOutOfRangeException 进行前置条件检查，不返回错误码或可空值。
- 测试断言：测试项目自带一个极简断言库 Assert，所有断言失败均抛出自定义 AssertionException；测试运行器 TestRunner 捕获 Exception 并记录为失败。
- UI 层：主窗体 MainForm 中未发现全局 try/catch 包裹事件回调，也没有自定义异常类型或错误码枚举，UI 仅负责转发用户输入到业务层，不自行处理业务错误。
- 无中间件统一入口：WinForms 应用没有类似 ASP.NET Core 的错误中间件，也未在 Program.cs 中注册全局异常处理器。

## 2. 关键文件与位置
- src/XyzController/Logic/AxisController.cs：构造时校验 max > min，抛 ArgumentException。
- src/XyzController/Logic/AxisJogService.cs：对 axis 参数抛 ArgumentNullException，对 step 和 direction 抛 ArgumentOutOfRangeException。
- src/XyzController.Tests/Testing/Assert.cs：自定义断言库，定义 AssertionException，提供 Throws<T> 辅助方法用于验证预期异常。
- src/XyzController/MainForm.cs：UI 层无显式 try/catch，错误通过异常向上传播或由调用方（测试）感知。

## 3. 架构与约定
- 分层清晰：UI（MainForm）→ 业务（XyzControllerHub / AxisController / AxisJogService）→ 控件（Controls）。错误仅在业务层以标准异常形式暴露，UI 层不吞异常也不做转换。
- 不可变范围加自动钳位：AxisController.SetTarget 内部对越界值执行 Clamp 而非抛错，属于正常边界处理；只有非法构造参数才抛异常，体现数据合法但越界是业务状态、参数非法才是错误的约定。
- 测试驱动的错误可见性：通过 Assert.Throws<T> 显式断言业务方法会抛出的异常类型，确保参数校验契约被测试覆盖。

## 4. 开发者应遵守的规则
1. 参数校验优先抛标准异常：在公共 API 入口处对 null、越界、非法枚举值等使用 ArgumentNullException、ArgumentOutOfRangeException、ArgumentException，不要返回 null 或错误码。
2. 业务边界用钳位而非异常：对输入合法但超出物理范围的情况（如目标值超出轴行程），应在方法内部 Clamp 后静默修正，不抛异常。
3. UI 层不捕获业务异常：保持 MainForm 作为薄壳，只转发事件；如需展示错误信息，应通过事件或属性通知上层，而不是 try/catch 后弹窗。
4. 测试中显式断言异常：使用 Assert.Throws<T> 验证前置条件，避免隐式崩溃导致测试误判。
5. 禁止使用 throw new Exception(...) 裸异常：始终选择语义更具体的派生异常类型，便于调用方区分处理。
