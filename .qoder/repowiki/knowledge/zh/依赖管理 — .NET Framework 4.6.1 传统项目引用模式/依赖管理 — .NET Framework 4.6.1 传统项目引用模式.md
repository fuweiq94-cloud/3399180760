## 系统概述
本仓库基于 .NET Framework 4.6.1，采用 Visual Studio 2017 兼容的传统（非 SDK 风格）`.csproj` + `.sln` 方案组织方式。依赖管理完全由 MSBuild 与 Visual Studio 内置的 `ProjectReference` 机制驱动，**未使用 NuGet、包管理器或第三方依赖工具**。

## 关键文件与结构
- `XyzController.sln`：解决方案入口，声明四个项目及其 GUID、配置映射。
- `src/XyzController/XyzController.csproj`：主 WinForms 业务库，通过 `<ProjectReference>` 引用 Controls 与 WpfHost。
- `src/XyzController.Controls/XyzController.Controls.csproj`：自定义控件类库，仅引用 System / System.Drawing / System.Windows.Forms 等框架程序集。
- `src/XyzController.WpfHost/XyzController.WpfHost.csproj`：WPF 宿主桥接层，同时产出 DLL 与独立 EXE（后置构建目标 `BuildExe` 调用 `Csc` 编译 `Bootstrapper.cs`）。
- `src/XyzController.Tests/XyzController.Tests.csproj`：自托管控制台测试项目，引用主项目，不依赖任何测试框架。

## 架构与约定
- **纯框架程序集引用**：所有 `<Reference Include="System.*" />` 均指向 .NET Framework 自带程序集，无外部 NuGet 包。
- **项目间依赖方向单向**：
  - `XyzController → XyzController.Controls`（控件库）
  - `XyzController → XyzController.WpfHost`（WPF 宿主接口）
  - `XyzController.Tests → XyzController`（测试）
  - WpfHost 不反向引用业务项目，避免循环依赖。
- **双输出策略**：WpfHost 项目通过 MSBuild `<Target Name="BuildExe">` 在 `AfterTargets="Build"` 阶段用 `Csc` 单独编译 `Bootstrapper.cs` 为 `XyzController.WpfHost.Launcher.exe`，实现“DLL 对外暴露接口 + EXE 独立启动”的组合产物。
- **版本锁定**：统一 `TargetFrameworkVersion=v4.6.1`、`ToolsVersion=14.0`（VS2015/2017）、`WarningLevel=4`（受 VS2017 编译器限制），并通过 `Deterministic=true` 保证可重复构建。
- **无锁文件/无私有源**：不存在 `packages.config`、`NuGet.Config`、`global.json` 等文件，也不存在 vendor 目录；所有依赖随 .NET Framework SDK 安装。

## 开发者应遵循的规则
1. **新增第三方库时**：优先评估是否必须引入；若必须，应在对应项目的 `<ItemGroup><Reference ... /></ItemGroup>` 中显式添加，并同步更新 `XyzController.sln` 中的项目顺序以保持依赖方向清晰。
2. **禁止跨项目循环引用**：当前 WpfHost 刻意不引用业务项目以维持单向依赖，新增模块需遵守同一原则。
3. **保持框架版本一致**：所有 `.csproj` 的 `TargetFrameworkVersion` 必须同为 `v4.6.1`，否则会导致运行时绑定失败。
4. **警告级别约束**：由于 VS2017 编译器限制，`WarningLevel` 不得超过 4，注释中已明确标注 CS1900 错误场景。
5. **测试项目零外部依赖**：测试运行器自实现于 `Testing/` 目录下，不引入 NUnit/xUnit，便于双击 exe 直接运行。

## 置信度说明
该仓库是一个小型桌面应用，依赖极其简单（仅 .NET Framework 框架程序集 + 内部项目引用），未发现任何 NuGet、包管理或私有注册表相关代码，因此对“现代依赖管理系统”的适用性较低，但上述分析完整覆盖了其实际采用的传统 MSBuild ProjectReference 模式。
