## 第一部分：先搞懂几个概念（术语扫盲）

别急着写代码，先花 5 分钟理解下面几个词，后面看代码会轻松很多。

### 1.1 什么是「上位机平台」？
**上位机**就是「上面那台机器/软件」——在工业现场，它通常指**主控电脑上的主程序**，负责：
- 显示总界面、调度各个功能模块
- 跟硬件（运动控制卡、PLC）打交道
- 管理项目、报警、权限等

而**工艺模组**（本类库）是「插在上位机上的功能插件」。上位机是「主机」，工艺模组是「插件」。

### 1.2 什么是「工艺模组」？
一个**工艺模组 = 一个独立的功能页面 + 它背后的控制逻辑**。本项目内置了 3 个：

| 模组 | 大白话 | 界面上能干什么 |
|------|--------|----------------|
| **MainControl**（主控制） | 手动操控盘 | 拖滑块、按 JOG 按钮、急停，手动移动 X/Y/Z 轴 |
| **PointJump**（点位跳转） | 收藏夹跳转 | 把常用位置存成「预设点」，点一下自动跳过去 |
| **TrajectoryView**（轨迹查看） | 运动轨迹记录仪 | 显示并记录 XYZ 走过的路径 |

每个模组都有**两张脸**：
- **运行界面 RunForm**：平时操作的主界面
- **设置界面 ModuleSettingForm**：改参数用的（像属性表格 PropertyGrid）

### 1.3 什么是「接口 / 依赖倒置」？（插座类比）

这是整个架构的**核心思想**，务必理解。

打个比方：
> 你家的**插座**（接口）是标准的。不管你插的是电视、冰箱还是手机充电器（具体实现），只要插头符合插座标准，就能通电。
> 墙里的电线（平台）**不需要知道**你插的是什么电器——它只负责按标准供电。

本项目里：
- **插座（接口）** = `IMotionService`（一套「运动控制」的标准约定）
- **电器（实现）** = 具体的硬件服务（比如某品牌控制卡、某型号 PLC）
- **墙里电线（平台/模组）** = 只认「插座」，不认「具体电器」

**好处**：将来换硬件（换「电器」），只要新硬件也实现 `IMotionService` 这个「插座标准」，平台和模组**一行代码都不用改**。这就叫「依赖倒置」——大家只依赖抽象标准，不依赖具体东西。

### 1.4 几个核心类的角色（餐厅类比）

把整个系统想象成一家**餐厅**：

| 类名 | 餐厅角色 | 它负责什么 |
|------|----------|------------|
| `ProcessModuleManager` | **前台总管** | 你（平台）只跟它打交道。它负责登记、安排、通知所有模组 |
| `ProcessModuleBase` | **员工手册** | 规定每个模组「必须会做什么」（初始化、显示界面、执行命令…） |
| `XyzControllerHub` | **模组的大脑** | 每个模组内部都有一个，负责算坐标、发指令 |
| `IMotionService` | **翻译官** | 把模组的「标准指令」翻译成「具体硬件听得懂的话」 |
| `RunForm` / `ModuleSettingForm` | **模组的脸** | 显示给用户看的界面 |
| `ProcessModuleEnvironment` | **广播站** | 广播「换项目了」之类的消息给所有模组 |

> 记住一句话：**平台主要跟「前台总管」`ProcessModuleManager` 说话就行**，不用直接指挥每个模组。

## 第二部分：整体流程，一图看懂

### 2.1 启动流程（平台要做的事）

```
平台启动
   │
   ├─ 第1步：连接硬件后端（创建 IMotionService 并 Open）
   │
   ├─ 第2步：初始化所有模组（ProcessModuleManager.InitAll）
   │
   ├─ 第3步：把硬件服务「插」进所有模组（InjectServiceToAll）★关键
   │
   ├─ 第4步：接好报警线（SubscribeAlarms）
   │
   ├─ 第5步：把模组界面显示到面板上（ShowRunForm）
   │
   └─ 第6步：打开项目（SwitchProject，可选）
```

### 2.2 数据怎么流动（运行起来之后）

```
用户在界面拖滑块
      │
      ▼
模组大脑 XyzControllerHub.SetTarget(目标坐标)
      │ 把目标打包成「标准指令」MotionCommand
      ▼
翻译官 IMotionService.ExecuteCommand(指令)
      │ 翻译成硬件听得懂的话
      ▼
真实硬件（控制卡/PLC）开始运动
      │ 硬件实时反馈「我现在到哪了」
      ▼
翻译官 IMotionService.PositionUpdated 事件（推送真实位置）
      │
      ▼
模组大脑更新当前坐标 → 界面刷新显示
```

> 注意：界面**从不直接碰硬件**，一切都经过「大脑」和「翻译官」。这样换硬件时界面不用改。

## 第三部分：手把手六步启动

下面每一步都按「干什么 → 为什么 → 代码」来讲。

### 第 1 步：连接硬件后端

**干什么**：创建一个「翻译官」（`IMotionService` 的实现），并打开它和硬件的连接。

**为什么**：模组发出的指令需要有人翻译成硬件语言；硬件的位置反馈也需要有人传回来。这个「翻译官」就是 `IMotionService`。

**怎么做**：
```csharp
// 类库已提供一个「翻译官骨架」PlatformMotionService，
// 真实项目里你要复制它、填入你硬件的 SDK 调用（见接口文档.md 第5节）。
IMotionService motionService = new PlatformMotionService();

// 打开连接（内部会初始化控制卡/建立PLC通信）
if (!motionService.Open())
{
    // Open 返回 false 说明连不上硬件，要处理（提示用户/重试）
    MessageBox.Show("运动控制后端连接失败，请检查硬件");
    return; // 连不上就别往下走了
}
```

> 💡 **没有真实硬件怎么办？** 见 FAQ 第 4 条。

---

### 第 2 步：初始化所有模组

**干什么**：让「前台总管」把内置的 3 个模组都登记并初始化好。

**为什么**：模组初始化时会读取自己的参数文件、准备好内部状态。必须先初始化，后面才能用。

**怎么做**：
```csharp
// InitAll 会一次性创建并初始化 MainControl / PointJump / TrajectoryView 三个模组
bool ok = ProcessModuleManager.InitAll();

if (!ok)
{
    // 有模组初始化失败（通常是参数文件损坏）
    MessageBox.Show("部分工艺模组初始化失败，请检查参数文件");
}
```

> 💡 `InitAll()` 是**幂等**的——意思是「调多次和调一次效果一样」，不小心调两次也不会出错。

---

### 第 3 步：把硬件服务「插」进所有模组（★最关键）

**干什么**：把第 1 步的「翻译官」交给每一个模组。

**为什么**：模组初始化时，它的大脑（Hub）里「翻译官」位置是**空的**（`null`），此时界面能动但**硬件不会动**。只有把翻译官插进去，指令才能真正发到硬件。**这一步漏了，就会出现「界面有反应、机器不动」的现象。**

**怎么做**：
```csharp
// 一行代码，把服务注入全部模组（内部会逐个调用模组的 SetMotionService）
ProcessModuleManager.InjectServiceToAll(motionService);
```

> 💡 就这一行，三个模组全打通。这就是「前台总管」的便利——你不用一个个去设置。

---

### 第 4 步：接好报警线

**干什么**：订阅所有模组的「报警事件」，这样模组出错时平台能收到通知。

**为什么**：模组执行命令出错（比如命令写错、参数不对）时会发出报警。如果平台不订阅，这些报警就「石沉大海」，你都不知道出错了。

**怎么做**：
```csharp
// 订阅：所有模组的报警都会调用 OnModuleAlarm 这个方法
ProcessModuleManager.SubscribeAlarms(OnModuleAlarm);

// 这是平台自己写的「报警处理」方法
private void OnModuleAlarm(object sender, ModuleAlarmEventArgs e)
{
    // e.ModuleName = 哪个模组报的警（如 "PointJump"）
    // e.Message    = 报警内容（如 "未知命令:XXX"）
    // e.Time       = 报警时间
    // TODO: 在这里把报警显示到平台的报警列表/弹窗
    Console.WriteLine("【报警】{0}：{1}（{2}）", e.ModuleName, e.Message, e.Time);
}
```

---

### 第 5 步：把模组界面显示到面板上

**干什么**：在你的平台界面上放一个 `Panel`（面板容器），然后让模组把自己的界面「填」进去。

**为什么**：模组的界面（RunForm）需要有个「容器」来显示。平台提供 Panel，模组自动铺满它。

**怎么做**：
```csharp
// 1) 先拿到主控制模组（用注册名 "MainControl"）
ProcessModuleBase main = ProcessModuleManager.Get("MainControl");

// 2) 确认它初始化成功了
if (main != null && main.bInitOK)
{
    // 3) 把它的运行界面显示到 panelHost 这个面板里
    //    （panelHost 是你平台界面上的一个 Panel 控件）
    main.ShowRunForm(panelHost);
}
```

> 💡 `ShowRunForm` 内部会自动把界面设成「铺满整个面板、不能拖动」，你不用管细节。
> 💡 想显示设置界面就调 `main.ShowSettingForm(另一个Panel)`。

---

### 第 6 步：打开项目（可选）

**干什么**：告诉所有模组「现在用哪个项目」，模组会自动加载该项目的参数。

**为什么**：不同产品/工件有不同的参数（比如不同的预设点位）。切换项目时，模组要重新读参数。

**怎么做**：
```csharp
// 切换到指定项目路径（模组会自动重载各自的项目参数）
ProcessModuleEnvironment.SwitchProject(@"D:\Projects\工件A");

// 或者用默认项目路径
ProcessModuleEnvironment.SwitchProject(ProcessModuleEnvironment.CurrentProjectPath);
```

## 第四部分：完整可复制的示例代码

把上面六步串起来，就是一个「平台启动器」。你可以**直接复制**这个类到平台里用：

```csharp
using System;
using System.Windows.Forms;
using InterfaceDefine;     // ProcessModuleBase、ProcessModuleEnvironment 在这里
using MainModule;          // ModuleAlarmEventArgs（报警参数）在这里
using ProcessModules;      // ProcessModuleManager、IMotionService、PlatformMotionService 在这里

/// <summary>
/// 上位机平台「工艺模组启动器」。
/// 平台启动时调 Start()，关闭时调 Shutdown()。
/// </summary>
public class PlatformBootstrapper
{
    private IMotionService _motionService;   // 保存翻译官引用，关闭时要用

    /// <summary>
    /// 平台启动时调用。
    /// runPanel    = 平台上用来显示「运行界面」的 Panel
    /// settingPanel= 平台上用来显示「设置界面」的 Panel（可传 null 暂不显示）
    /// </summary>
    public bool Start(Panel runPanel, Panel settingPanel)
    {
        // —— 第1步：连接硬件后端 ——
        _motionService = new PlatformMotionService();   // TODO: 换成你硬件的真实实现
        if (!_motionService.Open())
            return false;                               // 连不上，启动失败

        // —— 第2步：初始化所有模组 ——
        if (!ProcessModuleManager.InitAll())
            return false;                               // 有模组初始化失败

        // —— 第3步：注入硬件服务（一行打通全部模组）★ ——
        ProcessModuleManager.InjectServiceToAll(_motionService);

        // —— 第4步：接报警 ——
        ProcessModuleManager.SubscribeAlarms(OnModuleAlarm);

        // —— 第5步：显示主控制模组的运行界面 ——
        ProcessModuleBase main = ProcessModuleManager.Get("MainControl");
        if (main != null && main.bInitOK && runPanel != null)
            main.ShowRunForm(runPanel);

        // —— 第6步：打开默认项目（可选）——
        ProcessModuleEnvironment.SwitchProject(
            ProcessModuleEnvironment.CurrentProjectPath);

        return true;   // 启动成功
    }

    /// <summary>平台关闭时调用。</summary>
    public void Shutdown()
    {
        // 关闭顺序很重要：先退订报警 → 保存 → 关模组 → 关硬件
        ProcessModuleManager.UnsubscribeAlarms(OnModuleAlarm);
        ProcessModuleManager.SaveAll();      // 保存参数
        ProcessModuleManager.CloseAll();     // 关闭所有模组
        if (_motionService != null)
            _motionService.CloseService();   // 最后关硬件连接
    }

    /// <summary>报警处理：模组出错时这里会被调用。</summary>
    private void OnModuleAlarm(object sender, ModuleAlarmEventArgs e)
    {
        // TODO: 转发到你平台的报警系统（弹窗/列表/日志）
        Console.WriteLine("【报警】{0}：{1}（{2}）", e.ModuleName, e.Message, e.Time);
    }
}
```

**在你的平台主窗体里这样用**：
```csharp
public partial class PlatformMainForm : Form
{
    private PlatformBootstrapper _bootstrapper = new PlatformBootstrapper();

    private void PlatformMainForm_Load(object sender, EventArgs e)
    {
        // panelRun 是你界面上预先放好的一个 Panel 控件
        bool ok = _bootstrapper.Start(panelRun, null);
        if (!ok)
            MessageBox.Show("工艺模组启动失败");
    }

    private void PlatformMainForm_FormClosing(object sender, FormClosingEventArgs e)
    {
        _bootstrapper.Shutdown();
    }
}
```

## 第五部分：平台运行起来后能做什么

### 5.1 下发命令（让模组干活）

用模组的 `Action` 方法，第一个参数是「命令名」，后面是命令需要的参数：

```csharp
ProcessModuleBase pointJump = ProcessModuleManager.Get("PointJump");

// 让点位跳转模组跳到坐标 (10, 20, 5)
int ret = pointJump.Action("GOTO", 10f, 20f, 5f);

// 返回值含义：0=成功，-1=未知命令，-2=参数不对
if (ret != 0)
    Console.WriteLine("命令失败，错误码：" + ret);
```

### 5.2 三个模组各能干什么（命令速查表）

> 命令名**不区分大小写**（写 "goto" 或 "GOTO" 都行）。

**MainControl（主控制）**
| 命令 | 例子 | 作用 |
|------|------|------|
| `GOTO x y z` | `Action("GOTO", 10f, 20f, 5f)` | 三轴移动到指定坐标 |
| `ORIGIN` | `Action("ORIGIN")` | 回原点 (0,0,0) |
| `CENTER` | `Action("CENTER")` | 回到行程中心 |
| `JOG 轴 方向` | `Action("JOG", "X", 1)` | X 轴正向寸动一步（方向 +1/-1） |
| `SETSPEED 值` | `Action("SETSPEED", 80)` | 设速度档位（0~100） |
| `STOP` / `ESTOP` | `Action("STOP")` | 停止 / 急停 |

**PointJump（点位跳转）**
| 命令 | 例子 | 作用 |
|------|------|------|
| `GOTO x y z` | `Action("GOTO", 10f, 20f, 5f)` | 跳到指定坐标 |
| `GOTOPOINT 名字` | `Action("GOTOPOINT", "A工位")` | 跳到名为「A工位」的预设点 |
| `SAVEPOINT 名字` | `Action("SAVEPOINT", "P1")` | 把当前目标存为预设点「P1」 |
| `DELETEPOINT 名字` | `Action("DELETEPOINT", "P1")` | 删除预设点「P1」 |
| `SETSPEED 值` | `Action("SETSPEED", 60)` | 设速度档位 |
| `STOP` | `Action("STOP")` | 冻结所有轴 |

**TrajectoryView（轨迹查看）**
| 命令 | 例子 | 作用 |
|------|------|------|
| `GOTO x y z` | `Action("GOTO", 10f, 20f, 5f)` | 移动并记录轨迹 |
| `ORIGIN`/`CENTER`/`RANDOM` | `Action("RANDOM")` | 回原点/回中心/随机目标 |
| `CLEARTRAIL` | `Action("CLEARTRAIL")` | 清除轨迹 |
| `SHOWTRAIL on/off` | `Action("SHOWTRAIL", "on")` | 开/关轨迹显示 |
| `SETSPEED 值` | `Action("SETSPEED", 50)` | 设速度档位 |
| `STOP` | `Action("STOP")` | 冻结所有轴 |

### 5.3 读写参数

```csharp
ProcessModuleBase main = ProcessModuleManager.Get("MainControl");

main.SetParam("SpeedSetting", 80);        // 设置速度档位
object x = main.GetParam("CurrentX");     // 读取 X 轴当前位置
```

### 5.4 切换不同模组的界面

```csharp
// 想显示哪个模组，就 Get 哪个，再 ShowRunForm
ProcessModuleManager.Get("PointJump").ShowRunForm(panelRun);
// 或
ProcessModuleManager.Get("TrajectoryView").ShowRunForm(panelRun);
```

## 第六部分：如何正确关闭

关闭顺序很重要，**倒着来**（先关上层，再关硬件）：

```csharp
ProcessModuleManager.UnsubscribeAlarms(OnModuleAlarm);  // 1. 先退订报警
ProcessModuleManager.SaveAll();                          // 2. 保存参数
ProcessModuleManager.CloseAll();                         // 3. 关闭所有模组
motionService.CloseService();                            // 4. 最后关硬件连接
```

> 💡 顺序错了可能导致：报警重复触发、参数没保存、硬件资源没释放。

## 第七部分：常见问题 FAQ（小白最容易踩的坑）

**Q1：界面没显示出来？**
- 检查 `Get("MainControl")` 返回的是不是 `null`（名字拼错？没先 `InitAll`？）
- 检查模组的 `bInitOK` 是不是 `true`（初始化失败了？）
- 检查你传的 `Panel` 是不是真的在界面上、有没有尺寸（宽高为 0 就看不见）

**Q2：界面能动，但机器不动？**
- 99% 是**漏了第 3 步** `InjectServiceToAll`。模组大脑里没有「翻译官」，指令发不到硬件。
- 或者 `IMotionService.Open()` 没成功（`IsConnected` 为 false）。

**Q3：收不到报警？**
- 检查有没有调 `SubscribeAlarms(你的处理方法)`。
- 处理方法签名要对：`void 方法名(object sender, ModuleAlarmEventArgs e)`。

**Q4：没有真实硬件，怎么测试界面？**
- 可以先用一个「假翻译官」：复制 `PlatformMotionService`，在 `ExecuteCommand` 里把收到的指令对应的坐标，通过 `RaisePositionUpdated` 回推给前端，界面就会跟着动。
- ⚠️ 注意：这种「回环假服务」**只能用于本地调试界面**，正式对接必须用真实硬件反馈（项目规定严禁在生产中模拟位置）。

**Q5：编译报错「找不到 ProcessModuleManager / IMotionService」？**
- 检查文件顶部有没有这三行 `using`：
  ```csharp
  using InterfaceDefine;   // ProcessModuleBase、ProcessModuleEnvironment
  using MainModule;        // ModuleAlarmEventArgs
  using ProcessModules;    // ProcessModuleManager、IMotionService、PlatformMotionService
  ```
- 检查项目有没有**引用 `ProcessModules.dll`**（右键项目 → 添加引用）。

**Q6：怎么知道有哪些模组、它们叫什么名字？**
- 三个内置模组的注册名是固定的：`"MainControl"`、`"PointJump"`、`"TrajectoryView"`。
- 也可以遍历 `ProcessModuleManager.Modules` 拿到全部。

**Q7：能让两个模组同时显示吗？**
- 可以。给每个模组各准备一个 `Panel`，分别 `ShowRunForm` 即可（一个 Panel 同时只能放一个模组界面）。

### 8.2 平台集成 API 速查表

**ProcessModuleManager（前台总管）**
| 方法 | 作用 |
|------|------|
| `InitAll()` | 初始化全部内置模组 |
| `RegisterAndInit(模组, 名字)` | 注册并初始化单个（自定义）模组 |
| `Get(名字)` | 按名字拿模组 |
| `Modules` | 遍历所有模组 |
| `InjectServiceToAll(服务)` | 给所有模组注入硬件服务 ★ |
| `SubscribeAlarms(处理方法)` | 统一订阅报警 |
| `UnsubscribeAlarms(处理方法)` | 统一取消报警 |
| `StopAll()` / `SaveAll()` / `CloseAll()` | 统一停止 / 保存 / 关闭 |

**ProcessModuleEnvironment（广播站）**
| 方法/属性 | 作用 |
|------|------|
| `SwitchProject(路径)` | 切换项目并广播 |
| `CurrentProjectPath` | 当前项目路径 |
| `AppParamPath()` | 全局参数根目录 |

**ProcessModuleBase（每个模组都会的方法）**
| 方法 | 作用 |
|------|------|
| `ShowRunForm(面板)` | 显示运行界面 |
| `ShowSettingForm(面板)` | 显示设置界面 |
| `Action(命令, 参数...)` | 下发命令（返回 0=成功） |
| `SetParam(键, 值)` / `GetParam(键)` | 写 / 读参数 |
| `bInitOK` | 是否初始化成功 |

### 8.3 术语表

| 术语 | 解释 |
|------|------|
| 上位机 | 主控电脑上的主程序（平台） |
| 工艺模组 | 独立的功能插件（界面+控制逻辑） |
| 接口 (interface) | 一套标准约定（插座），具体实现可替换 |
| 依赖倒置 | 只依赖抽象标准，不依赖具体实现，方便替换 |
| 注入 (Inject) | 把某个对象「交给」另一个对象使用 |
| 事件 (event) | 「发生了某事」的通知机制（如报警） |
| 幂等 | 执行一次和执行多次效果相同 |
| DRO | 数字位置读数显示（界面上显示当前坐标的地方） |
| JOG | 点动：按住按钮轴就动，松开就停 |
